ROLE
You are on `devel` in the pfBlockerNG repo, performing Phase 2 of the trie
migration. Like Phase 1, this is STRICTLY BEHAVIOR-PRESERVING. You are
separating the PURE matching DECISION from the SIDE EFFECTS so the decision can
later be reimplemented on a trie without touching qstate/I/O code.

WHY THIS PHASE EXISTS
After Phase 1 the individual matchers are pure, but the orchestration that
chains them (data → zone → tld → idn → regex → whitelist, then hsts, then build
the block decision) is still inline in operate(), interleaved with
`dnsblDB[...] =`, `excludeDB.append(...)`, `write_sqlite(...)`, and DNSMessage
construction. To swap the data structure safely we need a single pure entry
point whose output can be compared old-vs-new by tests. This phase introduces
that entry point and a parallel one for the noAAAA path.
